# MOCS.DebugTool 产品使用说明书

## 一、产品概述

**MOCS.DebugTool** 是单悬浮架运营控制系统（MOCS）的配套调试工具，用于模拟外部车载设备（MCU / VCU）向 MOCS 系统发送 UDP 通信报文，以验证 MOCS 对各类报文的接收、解析、分发和状态更新逻辑是否正常工作。

**技术栈**: .NET 8 + Windows Forms + UDP Socket

---

## 二、产品功能列表

| 功能 | 说明 |
|------|------|
| **6 种报文类型模拟** | 支持模拟 MCU 状态/应答、EMS 状态A/B、VSPS 状态、OBC 共 6 类报文 |
| **双端口 UDP 发送** | 自动根据报文类型切换目标端口：6001（VCU）或 6002（MCU） |
| **自定义 UserData** | 支持十六进制格式输入 UserData，空格分隔或连续均可 |
| **随机报文快速测试** | 一键生成随机 UserData 并发送，无需手动填写 |
| **完整发送日志** | 终端风格深色日志面板，记录时间戳、目标端口、报文类型、序列号、长度、完整十六进制内容 |
| **窗口自适应缩放** | 所有控件随窗口大小自动缩放，支持任意窗口尺寸 |
| **序列号自增** | 每次发送报文 Seq 号自动 +1，模拟真实设备行为 |
| **错误提示** | 输入不合法时以红色字体在日志中显示错误原因 |

## 三、被测对象说明

DebugTool 通过发送符合 MOCS 协议规范的 UDP 报文，对 MOCS 系统的以下模块进行集成测试：

### 3.1 报文接收与解析

| 被测模块 | 说明 |
|----------|------|
| `UdpMessageService.ReceiveLoop()` | UDP 报文接收循环 |
| `MessageFactory.TryParseMessage()` | 帧头(0x02)/帧尾(0x03)/长度/CRC16 校验解析 |
| `MessageDispatcher.Dispatch()` | 基于报文类型的消息分发机制 |

### 3.2 MCU 侧业务逻辑（端口 6002）

| 被测模块 | 触发的处理入口 |
|----------|----------------|
| MCU 状态报文 (MsgId 0x81, 164B) | `MCUInterface.OnRecvMCUStatusMsg()` → 更新 `MCUStatus` 单例 |
| MCU 应答报文 (MsgId 0x82, 12B) | `MCUInterface.OnRecvMCUReplyMsg()` → 更新 `MCUReply` 数据 |

### 3.3 VCU 侧业务逻辑（端口 6001）

| 被测模块 | 触发的处理入口 |
|----------|----------------|
| EMS 状态A (CANID 0x21-0x34, 9B) | `VCUInterface.OnRecvEMSStatusMsgA()` → 更新 `EMSStatus` 单例 |
| EMS 状态B (CANID 0x61-0x74, 9B) | `VCUInterface.OnRecvEMSStatusMsgB()` → 更新 `EMSStatus` 单例 |
| VSPS 状态 (CANID 0x5F, 13B) | `VCUInterface.OnRecvVSPSStatusMsg()` → 更新 `VSPSInfo` 单例 |
| OBC 状态 (MsgId 0xF1, 16B) | `VCUInterface.OnRecvOBCStatusMsg()` → 更新 `OBCStatus` 单例 |

### 3.4 UI 数据刷新

| 被测窗体 | 订阅的数据模型 | 验证点 |
|----------|---------------|--------|
| `MOCS_UI`（主窗体） | MCUStatus, MOCSStatus, EMSStatus, VSPSInfo, OBCStatus | PropertyChanged 事件 → UI 实时刷新 |
| `MCU_UI` | MCUStatus | 牵引状态、故障状态显示 |
| `LCU_UI` | EMSStatus | 悬浮控制器各传感器数据 |
| `OBC_UI` | OBCStatus | 电池/电源状态 |
| `VSPS_UI` | VSPSInfo | 速度/位置数据 |

## 四、系统架构

```
[DebugTool GUI]  ──UDP──>  [MOCS 系统]
    │                           │
    │ 用户选择报文类型           │ UdpMessageService 接收
    │ 输入/随机 UserData         │ MessageFactory 解析
    │                           │ MessageDispatcher 分发
    │ MessageBuilder 构造帧      │
    │   ├─ BaseSendMsg           │ MCUInterface / VCUInterface 处理
    │   └─ MessageFactory        │
    │ UdpSender 发送             │ 单例数据模型更新
    │                           │
    │ LogSend 记录日志           │ UI 窗体 PropertyChanged 刷新
```

### 核心类说明

| 类 | 文件 | 职责 |
|----|------|------|
| `MainForm` | `MainForm.cs` | 主窗体，UI 布局、事件处理、日志输出 |
| `MessageBuilder` | `MessageBuilder.cs` | 报文构造器，创建 6 种符合协议的传输帧 |
| `UdpSender` | `UdpSender.cs` | UDP 发送器，向 localhost:6001/6002 发送数据 |
| `Program` | `Program.cs` | 应用程序入口 |

### 复用的 MOCS 核心类

| 类 | 用途 |
|----|------|
| `BaseSendMsg` | 报文主体（设置 MsgId、Seq、UserData 等） |
| `MessageFactory<BaseMessage>` | 将报文主体封装为传输帧（加帧头/帧尾/CRC16） |

## 五、操作指南

### 5.1 启动工具

```
方式一：在 Visual Studio 中将 MOCS.DebugTool 设为启动项目，按 F5 运行
方式二：编译后在输出目录直接运行 MOCS.DebugTool.exe
```

### 5.3 操作步骤

#### 方式一：发送自定义报文

1. 在 **"报文类型"** 下拉框中选择要模拟的报文类型
   - 目标端口会根据所选类型自动切换（MCU 类型 → 6002，其他 → 6001）
   - 也可手动修改目标端口
2. 在 **"UserData"** 输入框中填入十六进制数据，例如：
   ```
   01 02 03 04 05 06 07 08
   ```
   - 支持空格分隔或连续输入（如 `0102030405060708`）
   - 每种报文类型的 UserData 长度是固定的（见下表），输入错误会有提示
3. 点击 **"发送报文"** 按钮
4. 在 **"发送日志"** 区域查看发送结果

#### 方式二：发送随机报文

1. 选择报文类型
2. 直接点击 **"发送随机报文"** 按钮
3. 系统会自动生成符合长度的随机 UserData 并发送

### 5.4 报文类型与长度对照表

| 序号 | 报文类型 | MsgId | UserData 长度 | 目标端口 | 说明 |
|------|----------|-------|---------------|----------|------|
| 1 | MCUStatusMsg | 0x81 | **164 字节** | 6002 | MCU 状态消息（牵引状态/故障/位置/速度） |
| 2 | MCUReplyMsg | 0x82 | **12 字节** | 6002 | MCU 应答消息 |
| 3 | EMSStatusMsgA | 0x21-0x34 | **9 字节** | 6001 | EMS 状态消息 A 类（1 CANID + 8 CANData） |
| 4 | EMSStatusMsgB | 0x61-0x74 | **9 字节** | 6001 | EMS 状态消息 B 类（1 CANID + 8 CANData） |
| 5 | VSPSStatusMsg | 0x5F | **12 字节** | 6001 | VSPS 速度位置状态（额外 1 字节 CANID） |
| 6 | OBCMsg | 0xF1 | **16 字节** | 6001 | OBC 车载电池控制器状态 |

### 5.5 日志格式说明

每发送一条报文，日志区域会输出以下信息：

```
[HH:mm:ss.fff] → 目标端口  报文类型  Seq=序列号  Len=帧长度B
        完整帧十六进制内容
```

- **时间戳**：精确到毫秒
- **目标端口**：6001(VCU) 或 6002(MCU)
- **报文类型**：当前选择的报文类型名称
- **Seq**：报文序列号（从 1 开始自增）
- **Len**：完整传输帧的字节长度
- **错误日志**：以红色字体显示

## 七、注意事项

1. **MOCS 必须运行**：DebugTool 发送的报文由 MOCS 接收，如果 MOCS 未启动，报文将无法被处理
2. **本地回环通信**：默认使用 `127.0.0.1`，适用于本机调试
3. **端口一致性**：端口 6001/6002 必须与 MOCS 配置的监听端口一致
4. **UserData 长度**：每种报文类型有固定长度要求，输入长度不符会抛出异常
5. **序列号重置**：重新启动 DebugTool 后序列号从 1 重新开始
